\section{Background}
\label{background}

IRs (Intermediary Representations) are used across different stages of compilation 
in order to work at different levels of abstraction. This allows high level 
optimizations to be applied on a high level view of the code, while letting low 
level optimizations (like instruction selection) be handled close to the ISA. 

There is a lot of work needed to optimize these IRs but if programs can be proven equivalent 
(for example, by leveraging formal semantics) a technique arises called superoptimization. 
By enumerating all programs of a small size we may find a shorter program that is equivalent 
to a larger one  a larger one\textsuperscript{citation needed}. This technique can be applied 
across IRs, the target could even be an ISA. Therefore we can use superoptimization to find lowerings 
from a source IR (like LLVM IR) to an assembly language like RISC-V. Hence I'll refer to 
this as backend synthesis.

\section{Motivation}

Why go through the effort of providing formal semantics for your IR when you could just
go ahead and write the lowering yourself? Well, manually written backends are quite 
buggy. Additionally, with proper tools and support compiler developers will have an easier time
defining the semantics of their IRs.

Luckily, there exist verification tools that can prove the correctness of a lowering for 
which semantics are available \cite{smt-mlir}. However these require the compiler 
developer to both specify the semantics for their source and target representations, while 
also providing the lowering themselves. This is increases the overall amount of work 
required of the compiler developer.


The aim of this project is to give motivation for compiler developers to provide formal
bitvector semantics for their IRs by synthesizing a large portion of the backend. The
synthesized lowering will also verified so no additional work will be required for correctness.
Moreover, the synthesized lowering will use the minimal number of assembly instruction for
each IR instruction to achieve high performance.

Thus this project introduces a prototype of backend synthesis to a subset RISC-V, 
evaluating its effectiveness, as well as some optimization techniques that can be 
applied to speed up the process, achieving more coverage of the source representation,
and a more performant backend, all while being proven correct.

\section{Semantics}

At the base of the entire project sits the idea of formal semantics for our dialects.
We need to express how our instructions act on their inputs, and we want the semantics
to be easy to work with using automatic solvers. Therefore we use SMT formulas and model
our instructions as bitvector transformations. We also primarily care about basic arithmetic
operations so bitvector theory seems like a natural choice.

In order to remain within the confines of MLIR, we leverage the specialized SMT dialect

\subsection{Refinements}

Dialects don't always work on the same types of values (RISC-V dialect uses registers, and LLVM IR 
uses integers), checking if a lowering is ``correct'' isn't as easy as saying that for all inputs
the programs yield the same outputs. In fact we need to introduce the following \emph{refinement}
relation. A program is a \emph{refinement} of another if the following proposition holds:
\begin{equation*}
\text{\emph{target inputs refine the source inputs}} \implies \text{\emph{target outputs refine the source outputs}}
\end{equation*}

This way we can meanigfully talk about such lowerings. Notice that what it means for a value to 
\emph{refine} another is up to the implementation, however a clear example arises when considering 
poison in LLVM. Poison is a form of undefined behaviour that propagates, i.e. if any of the operands
of an operation are posion, than so is the result, while some operations generate poison (like an 
overflowing integer addition). This is useful because it can be used to optimize certain code sequences,
as we can always assume that poisoned values are convenable for optimization, since they are the
result of undefined behaviour. Looking at the following example, which checks if $x+1 > x$:

\begin{minted}{mlir}
llvm.func @f(%x: i32) -> i1 {
  %one = llvm.mlir.constant(1 : i32) : i32
  %add = "llvm.add"(%x, %one) {overflowFlags = #llvm.overflow<nsw>} 
      : (i32, i32) -> i32
  %cmp = llvm.icmp "sgt" %add, %x : i32
  llvm.return %cmp : i1
}
\end{minted}

Because of poison, this code can be optimized to, as we can ignore the undefined behaviour that
arises when $x = \texttt{INT\_MAX}$ by simply asserting that the poisoned value is \texttt{true}:
\begin{minted}{mlir}
llvm.func @f(%x: i32) -> i1 {
  %one = llvm.mlir.constant(1 : i1) : i1
  llvm.return %one : i1
}
\end{minted}

We can now see an useful refinement for poison: Any value \emph{refines} poison.

As mentioned in \ref{background}, superoptimization is a technique where a compiler 
attempts to find the optimal sequence of instructions that has the same semantics as a source
sequence. One technique to do this is via exhausting search over the program space and boolean 
tests to prove equivalence of the source and target.
%
It has been used to successfully find new optimizations both offline, and to improve
programs by running it as a compilation pass. Some very popular compilers have 
implemented and used this technique, like gnu superoptimizer for x86 assembly \cite{gnu-superopt},
and souper for LLVM IR \cite{souper}.



Lowering synthesis is a similar concept, but instead of searching for optimizations it searches
for valid lowerings between dialects. This is the novel idea behind the project, with the aim of
evaluating the efficiency and exploring the benefits of this technique.
%
The premise is simple, incur one large computational cost once for each source-target pair 
in order to search and find a safe lowering. This is obviously useful when compiling
to low level dialects that need to be lowered to various ISAs.
%
This reduces the number of bugs, but it also opens the door for the easy extension of the 
infrastructure with superoptimizers and other verification tools.

\section{Base RISC-V to SMT Semantics}

In order to lower a program written in the RISC-V dialect to its SMT semantics, we need to
be able to lower the types of all the primitive values (registers and immediates) together 
with all of the instructions as a sequence of SMT instructions. Thus we need to model for
the RISC-V assmenbly.


\subsection{Converting MLIR Types}

There are only a few types of values we will encounter in RISC-V code, given we are working 
with values in CPU registers, and encoded within instructions they can all be represented as 
bitvectors of apropriate sizes, however they will need to be extened to the same size for 
our smt operations.

\begin{itemize}
\item Register values are encoded in 64-bit bitvectors
\item Immediates are 12 bits, and should be extended to 64 bits
\item Shift values are 6 bits, and should be extended to 64 bits
\end{itemize}

\subsection{Handling diffrent Instruction Types}

RISC-V instructions can be divided into a few diffrent types, based on their structure.
Therefore we can extract the values, apply the appropriate conversion, then use the values
for the SMT semantics.

We have the following instruction types:
\begin{itemize}
\item Register-Register Operations

We can directly get the values of the registers and convert them to 64 bit bitvectors.

\item Register-Register Word Operation

Since we are using RV64, we also have some operations that are meant to work on a 32-bit word.

The semantics for this are the same as for Register-Register Operations, but the values are 
first truncated to 32 bits, and extended back at the end of the operation.

Example: 
\begin{minted}{mlir}
%op1 =  smt.bv.extract %rs1 from 0 : (!smt.bv<64>) -> !smt.bv<32>
%op2 =  smt.bv.extract %rs2 from 0 : (!smt.bv<64>) -> !smt.bv<32>
// Perform the operation on the truncated operands, yielding %tmp
%res = smt.bv.sign_extend %tmp : (!smt.bv<12>) -> !smt.bv<64>
\end{minted}

\item Register-Immediate Operation

We extract the immediate and extend it to a full sized bit vector, which can then operate 
with the register.

Example: 
\begin{minted}{mlir}
%op1 = %rs1
%op2 = smt.bv.sign_extend %imm : (!smt.bv<12>) -> !smt.bv<64>
// Perform the operation on the operands
\end{minted}

\item Register-Immediate Word Operation

We sign extend the immediate to 64 bits, than truncate the result, before sign extending it again.

Example: 
\begin{minted}{mlir}
%op1 = %rs1
%op2 = smt.bv.sign_extend %imm : (!smt.bv<12>) -> !smt.bv<64>
// Perform the operation on the operands, yielding %tmp
%tmp2 = smt.bv.extract %rs2 from 0 : (!smt.bv<64>) -> !smt.bv<32>
%res = smt.bv.sign_extend %tmp2 : (!smt.bv<32>) -> !smt.bv<64>
\end{minted}

\item Register-Register Shift (Word) Operation

We truncate the shift register (since only the lower bits matter), and then zero-extend it back 
to perform the operation.

Example for 64 bits:
\begin{minted}{mlir}
%op1 = %rs1
%tmp = smt.bv.extract %rs2 from 0 : (!smt.bv<64>) -> !smt.bv<6>
%op2 = smt.bv.zero_extend %tmp : (!smt.bv<6>) -> !smt.bv<64>
//Perform the operation, using op2 as the shift value
\end{minted}

For word operations we use 32 bit bitvectors, truncate rs1, and sign extend the result.

\item Register-Shamt Shift (Word) Operation

Extend the shift value from 6 bits the 64 (32) bits.

Example for 64 bits: 
\begin{minted}{mlir}
%op1 = %rs
%op2 = smt.bv.zero_extend %tmp : (!smt.bv<6>) -> !smt.bv<64>
//Perform the operation
\end{minted}

Again, word operations will have additional truncation of \texttt{\%rs} followed by sign extension of \texttt{\%res}.

\end{itemize}


\subsection{Exploiting identical semantics}

Across different instruction types we encounter instructions with the same underlying behaviour.
Take for example \texttt{riscv.add} and \texttt{riscv.addi}. Their behaviour can be expressed as a single
\texttt{smt.bv.add} opreration on the relevant values. So, as long as we are careful on how we extract
the values from each instruction we can exploit this similarity so have cleaner and more concise code.

Moreover, we also see how many semantics would only differ by the smt operation they perform.
Thus we recognize the following categories:

\begin{itemize}

\item Trivial Operations 
These operations can be expressed as a single smt operation.

\begin{minted}{mlir}
// obtain op1 and op2 as SSA Values
%res = smt.OPERATION %op1 %op2 : (!smt.bv<?>, !smt.bv<?>) -> !smt.bv<?>
// convert res to !smt.bv<64>
\end{minted}

The way we obtain the operands and process the result is defined by the instruction type, and
they type of \texttt{?} can be inferred from the types of the operands.

This is a the common case and includes a wide range of operations.


\item Comparisons

These require special care as the smt dialect comparison operations yield boolean values
that need to be converted into bitvectors. So semantics for comparison operations look like the
this:

\begin{minted}{mlir}
// obtain op1 and op2 as SSA Values
res_bool = smt.COMPARISON op1 op2 : (!smt.bv<?>, !smt.bv<?>) -> !smt.bv<?>
%zero = smt.bv.constant #smt.bv<0> : !smt.bv<64>
%on = smt.bv.constant #smt.bv<1> : !smt.bv<64>
%res = "smt.ite"(%0, %2, %1) :
    (!smt.bool, !smt.bv<64>, !smt.bv<64>) -> !smt.bv<64>
\end{minted}

\item Other

Some instructions have completely custom semantics, however with careful implementation
they plugged into the existing instruction types.

\end{itemize}

By building abstractions for these instruction types and common behaviours, we can very easily
implement the semantics for an instruction. This yields an easily extensible framework for 
defining semantics for RISC-V in the smt dialect.

\subsection{Leveraging xdsl-smt}

We use tools provided by xdsl and xdsl smt to create mlir blocks like the examples I've shown
above for every operation and SSA value. After that, I register my lowerings to xdsl-smt and
a lowering pass is created that recursively lowers all of the instructions from a source program
into a program using the smt dialect.

The smt program is then converted into a SMTLib S-expression, which will can be used by SMT solvers
in a variety of tools, in particular we care about validating the equivalence between different
such expressions.

\section{Theorems of equivalence in Lean}

In order to ensure the correctness of our semantics we programatically create lean4 theorems
to formally prove the equivalence of our semantics to those we consider a source of truth, which
are already written in lean.

We first obtain Lean functions that are representative of our RISC-V semantics, and we use them
as part of our equivalence theorems. First we encode our risc-v instructions into mlir blocks, 
with the arguments being the registers and immediates used by the instruction. Using our lowering to the smt dialect previously defined,
we first lower the block, obtaining the semantics of our instruction. 
%
All that's left is lowering from smt to a lean function, but since lean isn't an mlir
dialect we cannot use xdsl to our advantage when outputing lean code, so we also need to write 
working formatting for our lean functions and theorems.

\subsection{RISC-V to Lean Functions}

Walking through an example, we first start with a RISC-V instruction:
\begin{minted}{mlir}
%0 = riscv.slt %arg0, %arg1 : (!riscv.reg, !riscv.reg) -> !riscv.reg 
\end{minted}

Which gets lowered to semantics:
\begin{minted}{mlir}
%0 = "smt.bv.slt"(%arg0, %arg1) : 
   (!smt.bv<64>, !smt.bv<64>) -> !smt.bool
%1 = smt.bv.constant #smt.bv<0> : !smt.bv<64>
%2 = smt.bv.constant #smt.bv<1> : !smt.bv<64>
%3 = "smt.ite"(%0, %2, %1) : 
   (!smt.bool, !smt.bv<64>, !smt.bv<64>) -> !smt.bv<64>
\end{minted}

The next step is converting the semantics above into a lean function like the following:
\begin{minted}{lean}
let a0 : Bool := (BitVec.slt arg1 arg0) ;
let a1 : BitVec 64 := 0 ;
let a2 : BitVec 64 := 1 ;
let a3 : BitVec 64 := (cond a0 a2 a1) ;
\end{minted}

Thus our program is a series of let bindings for each operation and SSA value. We assign each
value a different ID and convert each element of the source program to a lean equivalent.
%
Types and attributes are straightforward, a \texttt{!smt.bv<X>} gets lowered to \texttt{BitVec X}
constant attributes get lowered directly, and their type is luckily inferred by the lean type checker.
Booleans behave in the same way.
%
We also have a variety of operations, some of which are grouped together when lowering so we can leverage
the similarity of their structure.

\subsubsection*{Constant Operation}
The source operation:
\begin{minted}{MLIR}
%res = smt.bv.constant #smt.bv<X> : !smt.bv<Y>
\end{minted}
is lowered to:
\begin{minted}{lean}
let res : BitVec Y := X ;
\end{minted}


\subsubsection*{Standard Binary Operation}
The source operation:
\begin{minted}{mlir}
%res = "smt.OP"(%arg0, %arg1) : (?, ?) -> ResultType
\end{minted}
is converted to:
\begin{minted}[escapeinside=||]{lean}
let res : |\textcolor{cyan}{ResultType}| := |\textcolor{cyan}{OP arg0 arg1}|;
\end{minted}
Where we use \textcolor{cyan}{cyan} to represent the lowered types, arguments and operation names.
%
Examples of binary operations are: \texttt{smt.bv.add, smt.bv.slt, smt.bv.ult} and more, all of which
have a lean equivelent like \texttt{BitVec.add, BitVec.slt, BitVec.ult}.

\subsubsection*{Shift Operation}
Lean bit vector shift operations use shift values of type \texttt{Nat} so we need to add a 
conversion for shift operations, so:
\begin{minted}{mlir}
%res = "smt.OP"(%arg0, %arg1) : (?, ?) -> ResultType
\end{minted}
is converted to:
\begin{minted}[escapeinside=||]{lean}
let res : |\textcolor{cyan}{ResultType}| := |\textcolor{cyan}{OP arg0}| (BitVec.toNat |\textcolor{cyan}{arg1}|);
\end{minted}

This pattern is used for operations \texttt{smt.bv.shl, smt.bv.lshr} and \texttt{smt.bv.ashr} which have
bitvector equivalents \texttt{BitVec.shiftLeft, BitVec.ushiftRight} and \texttt{BitVec.sshiftRight} respectively

\subsubsection*{Ternary Operation}
\begin{minted}{mlir}
%res = "smt.OP"(%arg0, %arg1, %arg2) : (?, ?, ?) -> ResultType
\end{minted}
is converted to:
\begin{minted}[escapeinside=||]{lean}
let res : |\textcolor{cyan}{ResultType}| := |\textcolor{cyan}{OP arg0 arg1 arg2}|;
\end{minted}

This pattern is used for operation \texttt{smt.ite} which has the
lean equivalent \texttt{cond}.

\subsubsection*{Extend Operation}

\begin{minted}{mlir}
%res = "smt.extendOP"(%arg) : (?) -> !smt.bv<X>
\end{minted}
is converted to:
\begin{minted}[escapeinside=||]{lean}
let res : |\textcolor{cyan}{ResultType}| := Bitvec.|\textcolor{cyan}{some}|Extend X |\textcolor{cyan}{arg}|;
\end{minted}
Where \emph{\texttt{start}} and \emph{\texttt{end}} are attributes of the source operation.

This pattern is used for operation \texttt{smt.bv.zeroExtend} and \texttt{smt.bv.signExtend}.

\subsubsection*{Extract Operation}

\begin{minted}{mlir}
%res = smt.bv.extract %arg from ? : (?) -> ResultType
\end{minted}
is converted to:
\begin{minted}[escapeinside=||]{lean}
let res : |\textcolor{cyan}{ResultType}| := BitVec.extractLsb |\emph{start} \emph{end} \textcolor{cyan}{arg}|;
\end{minted}
Where \emph{\texttt{start}} and \emph{\texttt{end}} are attributes of the source operation.

We use the patterns above to recursively lowers all operations from the semantics block into an 
appropriate Lean function.

\subsection{Generating theorems with bv\_decide}

After succesfully obtaining a lean function for the RISC-V instructions semantics we formulate 
equivalence theorems like the following, usinging the bv\_decide tactic, which leverages the 
bitblastable nature of the semantics and invokes a SAT solver to ensure that the theorem holds.

The theorems will take the same shape, first using the unfold tactic to replace the let bindings
with their underlying expressions, followed by bv\_decide, which finishes the proof:
\begin{minted}{lean}
theorem OP_eq (a0 : BitVec 64) (a1 : BitVec 64) 
    : RV64.OP a0 a1 = _OP a0 a1 := by
  unfold [RV64.OP, _OP]
  bv_decide
\end{minted}
Here, \texttt{\_OP} is the name of the function we automatically printed, and \texttt{RV64.OP} is
the functions which we consider a source of truth.
